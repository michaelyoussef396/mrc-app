### BUG-011 — `leads.lead_source` dual-vocabulary divergence; Zod validator is dead code

- **Surface:** `leads.lead_source` column — every write site
- **Severity:** 🟡 Major (silent data-quality issue; not user-visible at the lead level but breaks reports aggregation)
- **Wave:** 5 (deferred — architectural; requires Vryan input on source granularity)
- **Status:** Deferred to Wave 5

**Description:** `leads.lead_source` has two coexisting vocabularies persisted to the same column:

- **System-written values** (Framer EF default at `supabase/functions/receive-framer-lead/index.ts:689`): lowercase `'website'`. Also the only value the RLS policy `allow_public_insert_leads` (`supabase/migrations/20251111000020_allow_public_lead_creation.sql:24`) permits for anonymous INSERTs.
- **Admin-written values** (`CreateNewLeadModal.tsx:404`, `LeadDetail.tsx:1489`, `LeadBookingCard.tsx:153`): Title-Case strings sourced from `leadSourceOptions` (`src/lib/leadUtils.ts:102-124`) — `'Website Form'`, `'Google Search (Organic)'`, `'Google Ads'`, `'Customer Referral'`, `'Real Estate Agent'`, etc.

The Zod validator `leadSourceSchema` at `src/lib/validators/lead-creation.schemas.ts:166-177` defines a canonical lowercase vocabulary (`'website' | 'hipages' | 'google' | 'referral' | 'repeat' | 'facebook' | 'instagram' | 'other'`) but is **dead code** — zero runtime consumers in the production codebase. Grep confirms `leadSourceSchema`, `normalLeadSchema`, and `NormalLeadSchemaType` are referenced only inside the validator file itself. Every actual write path bypasses it.

**Investigation finding:**

- All four lead_source write sites mapped (Framer EF, CreateNewLeadModal, LeadDetail inline edit, Schedule LeadInfo edit). None pass through the Zod validator before INSERT/UPDATE.
- Zero `.eq('lead_source', '<value>')` query predicates exist in the application code today (grep clean), so the divergence is currently silent at the read side except in reports aggregation.
- Reports impact: `src/hooks/useReportsData.ts:96-104` source-config table only buckets lowercase keys (`'website'`, `'hipages'`, `'google'`, `'referral'`, `'repeat'`, `'facebook'`, `'instagram'`, `'other'`). Line 221 (`const source = lead.lead_source || 'other'`) drops every admin-created lead with a Title-Case value into the `'other'` bucket. The "Source breakdown" chart on `/admin/reports` is therefore inaccurate proportional to the share of admin-created leads.
- Future filtering, segmentation, or attribution work hits the same divergence — any predicate keyed on the lowercase vocabulary will miss the Title-Case rows.

**Why this didn't surface before:** the BUG-009 Schedule `<select>` rendering symptom is what exposed the underlying vocabulary divergence. Without that symptom, the issue would have remained invisible until the next reports-accuracy audit or attribution-keyed filter.

**Why Wave 4 (BUG-009 narrow fix) does not resolve this:** Wave 4 adds a first-class `{ label: 'Website', value: 'website' }` entry to `leadSourceOptions` so the canonical lowercase value renders correctly in the Schedule `<select>`. It does not reconcile the broader vocabulary divergence, does not wire the Zod validator into any write path, and does not backfill historical admin-created rows with Title-Case sources.

**Recommended fix scope (Wave 5):**

1. **Vocabulary decision (input from Vryan required — marketing/sales attribution granularity):** either
   - (a) align UI `leadSourceOptions` to the lowercase-terse vocabulary used by Framer EF + RLS + reports (collapse the 21 Title-Case options to the 8 Zod enum values), OR
   - (b) widen the canonical vocabulary to include the granular Title-Case values and update `useReportsData.ts` source-config + Zod enum + RLS policy comment to match.
2. **Wire the chosen vocabulary through every write site** (`CreateNewLeadModal`, `LeadDetail.saveField('lead_source', …)`, `LeadBookingCard.handleSaveLeadInfo`, `receive-framer-lead/index.ts` insert). Either consume the existing dead Zod validator or remove the validator and rely on `leadSourceOptions` as the single source of truth.
3. **Backfill historical rows:** `UPDATE leads SET lead_source = '<canonical>' WHERE lead_source IN (<Title-Case mismatches>)`. System-attributed migration using `SET LOCAL app.acting_user_id = SYSTEM_USER_UUID` (audit trigger captures the sentinel; Wave 3 `field_edit` rows are application-only and unaffected by raw SQL UPDATE).
4. **Delete the dead validator module** if no longer needed after the wiring decision in step 2.

**Action when picked up:** sequence the four steps above as a single Wave 5 PR. RLS policy (`allow_public_insert_leads`) constrains public/anonymous INSERTs to `lead_source = 'website'` and must remain authoritative for the Framer/public path regardless of which vocabulary direction is chosen.

#### Q12: 13% discount cap — DB-level enforcement CONFIRMED at two layers

Triggered by user's explicit ask. Grep of all migrations for `discount` + `CHECK`:

**Layer 1 — `inspections.discount_percent` CHECK constraint** (migration `20251122000001_add_pricing_columns_to_inspections.sql:127-128`):

```sql
ALTER TABLE inspections ADD COLUMN IF NOT EXISTS discount_percent NUMERIC
CHECK (discount_percent IS NULL OR (discount_percent >= 0 AND discount_percent <= 13));
```

Comment at `:130-131`: "Multi-day discount percentage applied. Range: 0-13%."

**Layer 2 — `invoices.discount_percentage` CHECK constraint** (migration `20260414000004_create_invoices_table.sql:46`):

```sql
CONSTRAINT discount_cap CHECK (discount_percentage >= 0 AND discount_percentage <= 13)
```

Migration header comment at `:3`: "13% discount cap enforced at both app layer AND DB (CHECK constraint)."

**Representation note:** both constraints use the percent scale (`<= 13`), NOT the decimal scale `<= 0.13`. The `pricing.ts` runtime uses the decimal scale (`MAX_DISCOUNT = 0.13`) for math, but persistence is percent. Anywhere the code converts decimal → percent for DB write (multiplication by 100), the cap is preserved. Anywhere it forgets to convert, an attempt to write `0.13` would pass the check trivially (0.13 ≤ 13), and `13` would still pass — so writes in decimal scale would silently appear as ~0% discounts in the DB. Worth a future audit of every writer of `inspections.discount_percent` / `invoices.discount_percentage` to confirm consistent scale.

**Cap bypass on manual override:** the CHECK constraint guards **only the `discount_percent` column**, not `total_inc_gst`. When `manualOverride = true`, pricing.ts writes `total_inc_gst = manualTotal` directly and does NOT touch `discount_percent` (which remains at its default 0 or null). The DB check passes (0 ≤ 13), but the **effective** discount % implied by `manualTotal` relative to the would-have-been tier total can be arbitrarily large. So:

- "13% cap is DB-enforced **when using the standard pricing flow**" — true ✅
- "Manual override bypasses the cap" — **partially true**: the bypass works not because the cap is missing but because the manual-override flow doesn't write to the gated column at all. The check never fires.

This is the precise shape of the gap. Worth flagging if/when the L1 equipment pricing audit is re-opened — the manual-override path needs an explicit policy decision: either (a) compute an effective `discount_percent` from `manualTotal vs auto-calc tier total` and write it (constraint will then fire if > 13%), or (b) document that manual override is a deliberate cap-bypass for admin-priced jobs and accept the audit gap.

#### Cross-surface inconsistencies + new BUG candidates

**Time formatter divergence — 4 implementations (already documented in v1):** LeadDetail / Schedule LeadBookingCard use `formatTimeForDisplay`; TechnicianJobDetail has an inline IIFE copy; Slack EF has its own inline copy; Schedule calendar surfaces use `formatTimeAU` for timestamptz (different input shape — not a bug, intentional divergence).

**Discount-percent scale divergence (BUG-019):** writer writes 0.13 decimal, DB column expects 0-13 percent, CHECK constraint accepts trivially, reader renders raw 0.13 → "0.13%" displayed instead of "13%".

**TypeScript types file stale (BUG-020):** `photos.deleted_at` (Stage 4.3) missing from `src/types/supabase.ts` AND `src/integrations/supabase/types.ts`. Last regen pre-dates the soft-delete migration.

**Manual-override column never persisted (BUG-021):** Section 9 `manualOverride` + `manualTotal` UI toggles write `total_inc_gst` directly (via pricing.ts manual-path bypass) but never set `manual_labour_override = true` or write the value to `manual_total_inc_gst`. The two boolean columns intended to flag manual overrides remain `null`/`false` even when a manual override was applied. Audit-trail gap: there is no DB-level signal that a given inspection's price was manually set rather than tier-calculated. Report aggregations / future analytics can't filter by manual-override status.

**Mould-boolean denormalisation orphan (BUG-022):** `inspection_areas` has 12 boolean columns (`mould_ceiling`, `mould_walls`, `mould_flooring`, etc.) that are NEVER WRITTEN by the current form. The form writes `mould_visible_locations` (jsonb array). Possible explanations: (a) legacy denormalisation that was deprecated but not dropped, (b) DB trigger expected to sync but none exists (grep clean), (c) pre-jsonb migration leftovers. Reports / queries filtering on these boolean columns would return zero matches. Confirm intent and either populate at save (sync writer) or drop columns in a future cleanup migration.

**`internal_moisture` PDF orphan (BUG-023):** PDF EF reads `{{internal_moisture}}` (mapped from `area.internal_moisture` column) but no programmatic writer populates it. The form's `area.moistureReadings[0]?.reading` (the internal reading) is written to `moisture_readings` table as a row, not to the `inspection_areas.internal_moisture` column. PDF renders empty/blank where the internal moisture % should appear.

**`extra_notes` admin-only orphan (BUG-024):** PDF EF reads `{{extra_notes}}` from `inspection_areas.extra_notes` (conditional on infrared photo presence at EF :1134) but tech form does not write this column. Either (a) admin writes it via EUF (`editable_fields` table — unverified), or (b) orphan-read. Run 6 risk: the extra_notes section on PDF area pages may render blank for tech-submitted inspections.

**`moisture_status` enum never set (BUG-025):** `moisture_readings.moisture_status` is a DB-enum-typed nullable column (`dry`/`elevated`/`wet`/etc.) intended to categorise the reading. Form writes only the raw `moisture_percentage`. No code path classifies the reading into the enum. Future-use column with no current writer; either drop or implement classification logic.

**Photo-to-moisture linkage race (BUG-026):** `photoUpload.ts` inserts photo with `moisture_reading_id = null`, then TIF :3502 retroactively updates the photo row's FK after the moisture_reading is saved. If anything fails between (transaction breakdown, network error, offline sync delay), the photo lands orphaned without its parent moisture_reading link. Stage 4.3 migration explicitly aligned this FK to `ON DELETE SET NULL` (not CASCADE) to govern lifecycle correctly, but the race still exists at upload time. Low frequency, but worth a Sentry breadcrumb to detect orphan photos in the wild.

#### BUG-019 — `discount_percent` scale mismatch between code (decimal) and DB (percent), silently mis-rendered

- **Surface:** All saved inspections with a multi-day discount; `InspectionDataDisplay.tsx:525-526` ("Discount: X%" line on admin lead-detail inspection view)
- **Severity:** 🔴 **HIGH** — silent data quality bug affecting reports + admin display + any future filtering on discount tier
- **Wave:** TBD — needs decision on fix direction (writer convert to percent? or migrate read sites to multiply by 100?)
- **Status:** pending
- **Investigation finding:**
  - `pricing.ts:29-35, 41` uses **decimal scale**: tier values are `0`, `0.075`, `0.1025`, `0.115`, `0.13`; `MAX_DISCOUNT = 0.13`.
  - `TechnicianInspectionForm.tsx:3362` writes `discount_percent: saveFullResult.discountPercent || 0` — directly persisting the decimal value to the DB column.
  - DB column constraint (migration `20251122000001:127-128`) expects **percent scale (0-13)**: `CHECK (discount_percent IS NULL OR (discount_percent >= 0 AND discount_percent <= 13))`. The constraint accepts `0.13` trivially (0.13 ≤ 13), so no DB error.
  - `InspectionDataDisplay.tsx:525-526` renders `${i.discount_percent}%` with no conversion — a 13% inspection displays as **"0.13%"** to admins.
  - `useVariationContext.ts:154-204` reads `discount_percent` for variation context — same value (0.13) flows through.
  - `invoices.discount_percentage` (migration `20260414000004:46`) — separate column, separate constraint, separate writer (`InvoicePaymentCard.tsx:202` writes `0`). The invoice column may be scale-correct; the inspection column is not.
- **Why this matters:** every saved inspection silently misrenders its multi-day discount. Reports aggregating by discount tier will see ~0% for everything. Future BI work that filters by discount-tier bucket will return empty. Customer-facing PDF doesn't display `discount_percent` directly (the discount amount flows through `total_inc_gst` correctly via `labourAfterDiscount` in pricing.ts), so customers are unaffected — but **internal admin views are silently lying**.
- **Fix options (require explicit decision before coding):**
  1. **Writer-side fix:** change `TechnicianInspectionForm.tsx:3362` to write `(saveFullResult.discountPercent || 0) * 100`. Single-line change. **One-time backfill required** for historical rows: `UPDATE inspections SET discount_percent = discount_percent * 100 WHERE discount_percent IS NOT NULL AND discount_percent <= 1`. The backfill predicate is safe (won't double-multiply rows already written correctly).
  2. **Reader-side fix:** change every read site to display `value * 100`. More invasive (multiple files), reverses scale convention versus what the DB constraint comment says ("Range: 0-13%"). Not recommended.
  3. **Schema migration:** change DB column comment + constraint to expect decimal scale (`<= 1`). Largest blast radius — affects invoices column too if symmetry is desired.

  **Recommendation:** Option 1 + backfill, with the constraint left as-is. Matches the documented intent ("Range: 0-13%") and is one-line writer fix + one-line backfill SQL.
