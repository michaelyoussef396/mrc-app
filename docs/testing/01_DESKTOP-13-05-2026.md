### BUG-007 ‚ÄĒ Activity log entries appear delayed
- **Surface:** `/admin/leads/<id>` ‚ÄĒ Activity timeline section
- **Repro:** Save a field ‚Üí activity row persists to DB immediately but UI doesn't show it for up to 30 seconds
- **Wave:** 3 (architectural)
- **Status:** pending fix (invalidate `['activity-timeline']` on every writer + drop polling interval to 5s safety net)

**Claude Code fix notes:**
- Root cause: `useActivityTimeline.ts` polled at `refetchInterval: 30000` (30s) with no Postgres realtime subscription. `useLeadUpdate` invalidated `['lead', leadId]` and `['leads']` but never `['activity-timeline']`. `handleChangeStatus` and `bookInspection` didn't invalidate either. Result: a save persisted instantly to DB but the UI didn't re-render until the next 30s polling tick.
- Fix: every Wave 3 writer now calls `queryClient.invalidateQueries({ queryKey: ['activity-timeline'] })` after a successful write (`useLeadUpdate`, `LeadDetail.handleChangeStatus`, `bookingService.bookInspection`). Polling interval reduced from 30000ms to 5000ms as a defence-in-depth safety net for cases where invalidation can't reach the timeline (e.g. background tab observing server-side activity row inserts).
- Files changed: `src/hooks/useActivityTimeline.ts` ¬∑ `src/hooks/useLeadUpdate.ts` ¬∑ `src/pages/LeadDetail.tsx` ¬∑ `src/lib/bookingService.ts`
- Commit: `d9aec0e`
- Verified by: Run 3 ‚úÖ ‚ÄĒ production smoke confirmed activity rows appear within ~1s of save (well under the 5s polling fallback), no perceptible delay
