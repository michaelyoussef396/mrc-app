### BUG-035 — Google Places autocomplete not working on Create New Lead modal — RESOLVED `9e8e1fc` (via BUG-039 fix)

- **Surface:** `/admin` → "Create New Lead" → Street Name & Number field (also broken on LeadDetail inline-edit address + Schedule booking sidebar — same `useGoogleMaps.ts` hook)
- **Repro (pre-fix):** Open the modal, type in Street field, dropdown does not appear
- **⚠️ RETRACTION — Phase 1 diagnosis was WRONG.** Wave 6.1 PR #2 labeled this Path (a) Cloud Console action. That was a misdiagnosis. The hook's wiring IS correct (correct call). The API key IS fine (confirmed post-PR-#2 by `mapsLoaded === true` rendering "Powered by Google Places" affordance in fresh Incognito). **Actual root cause: our `vercel.json` CSP `connect-src` directive did not allowlist `https://places.googleapis.com`, the host where the new Places API actually lives.** The XHR was blocked client-side; Cloud Console permissions were never the issue. See BUG-039 below.
- **Lesson logged:** for any "API not responding" diagnosis, first check Network tab + browser console for CSP/CORS errors. Do NOT default to Cloud Console without inspecting the client-side request status.
- **Severity:** MEDIUM (autocomplete broken on every admin surface; manual typing still worked)
- **Status:** **RESOLVED in production at `9e8e1fc`** via Wave 6.1 PR #3 (BUG-039 fix). Path (a) Cloud Console hypothesis fully retracted.

### BUG-039 — `vercel.json` CSP `connect-src` missing places.googleapis.com + fonts.googleapis.com — RESOLVED `9e8e1fc`

- **Surface:** `vercel.json` line 13, `connect-src` directive of the production CSP header
- **Repro (pre-fix):** Fresh Incognito → mrcsystem.com → any admin surface using `useAddressAutocomplete` → type address → DevTools Console shows:
  ```
  Connecting to 'https://places.googleapis.com/$rpc/google.maps.places.v1.Places/AutocompletePlaces'
  violates the following Content Security Policy directive:
  "connect-src 'self' https://...supabase.co wss://...supabase.co https://*.supabase.co
   https://*.sentry.io https://maps.googleapis.com"
  ```
- **Root cause:** The hook calls `google.maps.places.AutocompleteSuggestion.fetchAutocompleteSuggestions(...)` which hits `https://places.googleapis.com` — distinct from `https://maps.googleapis.com` (which loads the JS SDK and was the only Google host in `connect-src`). Same miss for `https://fonts.googleapis.com` (the service worker's `fetch()` to cache the Inter font CSS routes through `connect-src`, not `style-src`).
- **Fix shipped via Wave 6.1 PR #3 (`production` @ `9e8e1fc`, merged 2026-05-14):**
  - Added `https://places.googleapis.com` to `connect-src`
  - Added `https://fonts.googleapis.com` to `connect-src`
  - Specific hosts only, NO wildcards (per Michael's directive)
- **security-reviewer agent verdict on the CSP diff:** GREEN. All 7 checks PASS (wildcard discipline, scope creep, other CSP directives, other security headers, API key surface, SW scope, TLS). Risk delta vs prior CSP: effectively zero.
- **Verification post-deploy:** `curl https://mrcsystem.com` response header confirms `connect-src` now includes both new hosts. Manual Network tab check pending Michael's PWA-reset + Incognito smoke (autocomplete dropdown should appear, `places.googleapis.com` XHR should return 200).
- **Severity:** HIGH (broke Places everywhere; symptom looked exactly like Cloud Console issue, which caused PR #2's misdiagnosis)
- **Status:** RESOLVED in production at `9e8e1fc`. Manual smoke ownership: Michael.
