A ground-truth tour of the MRC operations system — what's built, what's deliberately not, and what's still a gap. Three role journeys, four cross-cutting references, five appendices. Written so three different readers (admin, co-owner, auditor) find what they need without context-switching.

★Day One — the four things that matter most

If you're new to the business or to the system, internalise these four before anything else. They explain ninety percent of the surprises a new admin will hit in their first week.

  • Phone calls fill every gap the app leaves. The app does not handle cancellation, dispute, payment chasing, or out-of-hours communication. When a customer wants to change something, they ring the office. The app records the result of that call after the fact.
  • Technicians learn about new jobs by opening the app. There is no push notification, no SMS, no email when a job is assigned. If you assign a tech to a job, you also need to text or call them — the app will not.
  • The 13% discount cap is enforced at the database. It is not a UI convention. The invoices table has a CHECK constraint that refuses any value above 13%. There is no override. Any pricing exception above 13% requires a code and database change.
  • Slack is the system's nervous system. Every status change, every booking, every payment, every overdue milestone posts to the team Slack channel. If something happened in the app and it didn't appear in Slack, it didn't really happen — start your investigation there.

1.2 What happens when a Framer submission arrives

✓ BUILT The Edge Function processes every webhook in this exact sequence:

  1. Raw body logged. Before any parsing, the raw payload is written to the webhook_submissions table with status 'received' (:429-435). This is the audit-survives-everything log: even if the function later fails, the raw payload is preserved for manual recovery.
  2. Body size + rate-limit checks. Bodies over 50 KB are rejected (413). IPs that have submitted more than 5 times in the last hour are rejected (429). Both update webhook_submissions.status to 'failed' or 'rate_limited' respectively (:11-12, 450-475) and fire admin alert email + Slack.
  3. Body parsed. Supports JSON, multipart form-data, and url-encoded content types (:483-499).
  4. Smart field extraction. Tries known field-name aliases first (e.g. accepts both full_name and "Full Name"); then falls back to regex content detection on remaining values (email pattern, phone pattern, date pattern, time pattern) for any unmatched fields. Robust against Framer field renames (:501-561).
  5. Zod validation. Failure inserts a 'failed' webhook_submissions row, fires admin alert email + Slack, returns 400 (:582-593).
  6. Duplicate check. Same email + phone within the last 24 hours returns 200 with "Lead already received" message and updates the submission status to 'duplicate' (:595-607).
  7. Insert with retry. Three attempts, exponential backoff (500 ms → 1500 ms → 4500 ms). Three failures triggers admin alert email + Slack and returns 500 (:364-383, 623-637).
  8. Fire-and-forget notifications. On success: Slack Block Kit message to the team channel ("🏠 New Lead Received"), confirmation email to the customer (branded MRC template, sent via Resend directly — does not go through the send-email Edge Function), email_logs row inserted with template name 'framer_lead_confirmation' (:653-698).

The submission-status column on webhook_submissions feeds the "Failed Webhooks" KPI on the Admin Dashboard. Any row with status 'failed' in the last 7 days surfaces there.

1.5 How the customer pays the invoice

DELIBERATE No payment processor integration, by design. MRC tracks payments as admin-entered data — method, date, and reference — but never processes them. Customers always pay by EFT outside the app. No Stripe, Square, or PayPal is planned; the five payment-method labels in code (cash, visa, mastercard, bank_transfer, cheque) exist so admin can record how a customer paid, not to represent an integration that needs building.

Once the money lands, an admin opens the customer record, clicks Mark as Paid, picks the method from a dropdown, enters the payment date and an optional reference. The invoice status moves to paid. The constraint payment_method IN ('cash','visa','mastercard','bank_transfer','cheque') is enforced both in the UI dropdown (src/components/leads/InvoicePaymentCard.tsx lines 551–557) and at the database level (supabase/migrations/20260414000004_create_invoices_table.sql line 32).

2.5 What blocks submission

The Zod schema enforces three hard requirements:

  1. Inspection Date must be set.
  2. At least one area must be present.
  3. At least one treatment method must be selected.
  4. At least some labour hours OR manual price override.

Photos are not required to submit. Moisture readings are not strictly required. If the technician misses required fields, errors highlight inline and the form scrolls to the first failing field.

2.6 Offline behaviour

✓ BUILT The system is built for spotty signal at job sites.

  • Auto-save interval — 30 seconds (constant in useJobCompletionForm.ts).
  • What survives offline — all form text (drafts in IndexedDB inspectionDrafts table) and photos (queued in photoQueue table after browser-side resize).
  • What the technician sees — an amber OfflineBanner at the top: "You're offline. X change(s) pending." Dismissible with X.
  • What a failed sync looks like — the photo is marked status='error' with the error message. Banner surfaces the count. Retry is implicit on the next online cycle.

Caveat: mutations (writes) in React Query are configured with retry: false. So if the tech is briefly online but the request fails for another reason (e.g. RLS rejection, timeout), they get an immediate error toast — no background retry.

4.1 What we don't notify (and why)

The following channels and events are silent — by intention, or by absence. Distinguish carefully:

Deliberately silent

  • DELIBERATE No SMS to anyone, ever — no Twilio, no MessageBird, no aggregator. Cost and AU compliance considerations.
  • DELIBERATE No customer-facing push or in-app messaging — customers never log in.
  • DELIBERATE No automated overdue communication to the customer — the office wants the human conversation when chasing money. See 1.6.
  • DELIBERATE No per-technician Slack DM — Slack is for the office channel; per-tech messaging would fragment the team's shared awareness.
  • DELIBERATE No automated customer invoice email. By design, the admin copies the Invoice Summary card data into their own email client and sends the invoice manually. Slack alert fires internally.

Accidentally silent (genuine gaps)

5.What Happens When Things Go Wrong

Real failure modes. For each: what the system does on its own, what a human has to do. Where something is missing, it's flagged loudly.

5.1 An email bounces

⚠ PARTIAL The email_logs table has status enum values for bounced, soft_bounce, spam, etc. — the schema is ready. But there is currently no Resend webhook listener implemented to flip those statuses automatically. In practice: the send call succeeds (status sent) and the system is unaware if Resend later marks it bounced. An admin would discover the bounce only by:

  • Looking at the Resend dashboard directly, or
  • Phoning the customer who never replied.

Human required: manually correct the customer's email in the lead and re-send the affected message via the lead detail email composer.

5.2 A customer disputes an invoice

✗ NOT BUILT No formal dispute workflow. The customer phones or emails. The admin can:

  • Edit the invoice (line items, total) if status ≠ paid.
  • Mark it void via the API.
  • Add a note to the activity log explaining the dispute.

There is no structured dispute reason, no comment thread, and no timer/SLA. The dispute history exists only as free-text activity entries. Risk: tribal knowledge, no consistency between admins.

5.3 Admin disagrees with a submitted inspection

⚠ PARTIAL Two paths:

  1. Edit in place — admin opens the inspection in adminMode, corrects fields, approves. Changes are diff-logged to the activity timeline.
  2. Send back — change the lead status to inspection_waiting. The technician sees a Revisions banner on their dashboard with the admin's note next time they open the app.

Communication gap: there is no push, email, or SMS to the technician when their work is sent back. The Revisions banner only appears the next time the technician opens the app — often hours later. Admins follow up by phone or text outside the app to close the loop.

5.4 A customer wants to cancel after booking

✗ NOT BUILT No customer-facing cancel button. The flow is:

  1. Customer phones MRC.
  2. Admin opens the schedule, finds the booking, changes its status.
  3. Admin manually emails a cancellation confirmation (or doesn't).
  4. Admin manually pings the technician (Slack, phone, text — outside the app).

No structured cancellation reason is captured. The activity log records the status change but not the why.

5.5 Payment doesn't arrive

✓ BUILT (with caveats) The cron flips the invoice to overdue the day after the due date. Slack alerts fire on a schedule (Day 15, 22, 29, 30, 60). There is no automated chase email to the customer; admin must phone or write manually. If the customer never pays, the invoice sits as overdue indefinitely. Escalation past 60 days is a manual, outside-the-system process (likely debt collection).

5.6 Photo upload fails mid-sync

✓ BUILT The photo's row in the IndexedDB photoQueue table is updated to status='error' with the error message. The OfflineBanner displays the count of pending + errored items. The technician can:

  • Wait for the next online cycle — sync retries automatically.
  • Move to better signal and the sync will pick up.
  • If the photo is permanently broken (corrupt, too large, quota exceeded), the only mitigation is to delete it from the queue (limited UI exposure today) and re-shoot.

5.7 The app throws an error at runtime

✓ BUILT Two layers of error boundary:

  • Global ErrorBoundary — catches anything that escapes the page level. Shows a full-page fallback with "Refresh Page" and "Go to Dashboard" buttons. No technical detail exposed to the user.
  • PageErrorBoundary — wraps high-risk pages (Inspection AI Review, Inspection Form, Job Completion Form, Admin Inspection Form, Lead Detail). Renders an inline error in just that section with "Try Again" and "Go Back" buttons; the rest of the app keeps working.

Errors are forwarded to Sentry with user role + URL context. Breadcrumbs of recent actions are attached. There is no in-app "Report this error" link; users don't know the error has been logged.

5.8 Database connection lost mid-request

React Query is configured with retry: 2 for reads (exponential backoff up to 10 s) and retry: false for writes. So a read failure briefly retries, but a write failure surfaces to the user as an immediate toast error with no background retry. The OfflineBanner appears when navigator.onLine goes false. On reconnect, the SyncManager drains the IndexedDB queue.

5.9 Live UI buttons that go nowhere

⚠ MISLEADING UI The "SMS" action button on the Lead Detail page (src/pages/LeadDetail.tsx:477) is wired to a handler called handleSMS() that has no implementation — clicking does nothing. Visible to admins, no error toast, no console message. Tracked as P1 in Section 11; the fix is either wire it to a Twilio path (which would change the DELIBERATE SMS decision) or remove the button. Until then, train new admins to use phone or email and ignore the SMS button.

5.10 No dev environment to test changes against

P0 INFRASTRUCTURE GAP. There is one Supabase project, ecyivrxjpsmjmexqatym, and it is production. Every migration, every Edge Function deploy, every schema experiment runs against the live database that customer data sits in. There is no preview, no staging, no rollback rehearsal. Mitigations today: migrations are reviewed manually before deployment; destructive operations require explicit approval per CLAUDE.md. Until a dev project is provisioned, every schema change carries the risk of a production-data incident with no rehearsal. Tracked as P0 in Section 11.

9.2 Functions and triggers

The following SQL functions and triggers enforce business rules at the database layer. They are the reason certain bugs are impossible: even if the UI is bypassed, the database refuses.

  • is_admin() and is_admin(uuid) — RLS helper used by every admin policy. Defined in 20260109000003_fix_is_admin_function.sql (replaced an earlier version that queried a since-dropped users table).
  • generate_invoice_number() + trigger set_invoice_number — auto-formats INV-YYYY-NNNN on INSERT. Sequence invoice_number_seq backs the counter. Hardened in 20260421000001_harden_generate_invoice_number.sql with an explicit search_path.
  • generate_inspection_number() — formats INS-YYYYMMDD-XXX with a daily reset (20251111000011, race-fix in 20251111000012).
  • generate_lead_number() + trigger add_lead_number — auto-numbers leads (20251112000003).
  • trigger_lead_activity() + trigger add_lead_activity — writes a row to activities on every lead INSERT/UPDATE (20251112000020). This is why the activity timeline on Lead Detail is always in sync.
  • check_booking_conflicts() and has_travel_time_conflict() — used by the booking modal before INSERT to prevent double-booking and to enforce the 15-minute buffer plus calculated travel time between consecutive jobs (20251111000011).
  • calculate_travel_time() + get_zone_by_suburb() — the SQL fallback for the Google Maps Distance Matrix call, backed by the suburb_zones 4×4 matrix (20251111000006).
  • audit_log_trigger(action_text) — SECURITY DEFINER function that writes old/new values to audit_logs. Triggered on leads, inspections, inspection_areas, user_roles (all of INSERT/UPDATE/DELETE, migration 20260311000001) and on invoices (INSERT/UPDATE only, migration 20260414000004).
  • prevent_audit_log_delete() + prevent_audit_log_update() — raise an exception on any attempt to modify audit_logs rows. The audit log is append-only by enforcement, not convention.
  • update_updated_at_column() — generic touch trigger on most tables.

10.The Integrations

Eight external services touch the system. For each: what it's used for, where the credentials live, what breaks if the service goes down, and who notices.

10.1 Supabase

The platform layer. Provides the Postgres database, authentication, file storage, Edge Functions runtime, and realtime subscriptions. One project: ecyivrxjpsmjmexqatym. Credentials: SUPABASE_URL and SUPABASE_SERVICE_ROLE_KEY in Edge Function secrets; VITE_SUPABASE_URL and VITE_SUPABASE_ANON_KEY in Vercel frontend environment. If Supabase goes down: the entire app is dark. Login fails, every query fails, the offline banner appears for online users. Technicians on the road can still fill out forms (data queues to IndexedDB) but sync is blocked until Supabase recovers. Sentry catches the failures; the team notices via the activity dashboards going stale.

10.2 Resend

Outbound transactional email. All branded customer emails — booking confirmations, report deliveries, Google review requests — go through Resend via the send-email Edge Function. Sender domain is mrcsystem.com as of commit cf2b3d8; the planned switch to mouldandrestoration.com.au is tracked in Section 11. Credentials: RESEND_API_KEY in Edge Function secrets. If Resend goes down: customer emails fail silently (caller does fire-and-forget). The email_logs table records the failure but no admin is paged. Technicians and admins notice only when a customer complains about not receiving a report. Bounces are a separate failure mode: see Section 5.1.

10.3 Slack

Office team notifications. Every status change, every booking, every payment, every overdue milestone posts to a single configured webhook channel via the send-slack-notification Edge Function (Block Kit format). Credentials: SLACK_WEBHOOK_URL in Edge Function secrets. If Slack goes down or the webhook URL becomes invalid: Slack posts fail silently. The team loses situational awareness — new leads don't pop up in chat, payments aren't visible — and will only notice when someone checks the admin dashboard expecting recent activity. The system itself continues to operate normally; only the team's nervous-system feed is interrupted.

10.4 Google Maps

Distance Matrix API for travel-time calculation between technician locations and job sites. Used by calculate-travel-time Edge Function and consumed by the booking modals to suggest available dates and detect travel-buffer conflicts. Credentials: GOOGLE_MAPS_API_KEY in Edge Function secrets. If Google Maps is unreachable: the booking modal shows a warning and the admin can override; the database has a fallback calculate_travel_time() SQL function backed by the suburb_zones 4×4 zone matrix (CBD / Inner / Middle / Outer). Bookings still work; travel-time accuracy degrades. The admin notices through inaccurate travel estimates on the schedule view.

10.5 OpenRouter (Gemini)

AI inspection summary generation. Called by generate-inspection-summary with a 3-model fallback chain (primary, then two backups) so a single model going offline doesn't break the feature. Credentials: OPENROUTER_API_KEY in Edge Function secrets. If OpenRouter is completely down or all three models fail: the AI Review screen shows an error; the admin can retry or write the summary by hand. The lead doesn't progress until the summary exists. The admin notices immediately.

10.6 Sentry

Application-side error tracking and performance monitoring. Wraps the React tree (src/components/ErrorBoundary.tsx, PageErrorBoundary), captures uncaught exceptions with breadcrumbs and user role context, and attaches replay snapshots. Configured via VITE_SENTRY_DSN at runtime; source-map upload uses build-time variables. If Sentry is down: the app continues working; the team simply loses the error feed. Errors still bubble up to the user via error boundaries. New issues won't be visible in the Sentry dashboard until it recovers.

10.7 Vercel

Hosting and CDN. Auto-deploys the React app from the main branch to preview URLs and from the production branch to the live URL. Enforces security headers (CSP, X-Frame-Options, Referrer-Policy). Frontend environment variables (VITE_*) are managed in the Vercel project settings. If Vercel goes down: the app is unreachable for everyone. Edge Functions and the database remain online (separate Supabase infrastructure) but no UI can reach them. The team notices instantly — nobody can log in.

10.8 Framer

The public marketing/booking website. Submits new leads to the receive-framer-lead Edge Function as webhook POST. The Edge Function smart-extracts fields and creates a lead. Credentials: none on the MRC side — Framer calls a public endpoint (rate-limited 5/hr/IP). If Framer goes down: the website is offline; nobody can submit a new request. Existing leads, the admin app, and the technician app are unaffected. Marketing notices first.

10.9 hipages — not an integration

hipages_lead appears in the lead status enum, suggesting integration. It is not. There is no API call, no webhook handler, no scheduled poll. "Hipages" is a manual flag an admin sets when a lead originated from hipages.com.au. Documented here because new readers reasonably assume otherwise.
